## 📖 第一部分：项目概述

### 项目定位

STM32F103C8T6 SPL库快速开发工程模板，基于board.h硬件隔离和modules分层架构，遵循防御性编程与AUTOSAR分层思想。

**使用说明（给AI）：**
1. 新建会话时粘贴"项目概述"段落作为开场
2. 直接说明新模块需求（如："添加PWM模块驱动电机"）
3. 自动遵循现有代码规范与架构风格
4. 无需重复解释项目结构或历史问题

### 核心特性

- **硬件抽象**：BSP层隔离，board.h配置表驱动，跨型号移植便捷
- **模块化架构**：清晰的分层设计（BSP/驱动/系统/应用），职责分离
- **统一错误处理**：错误码系统、错误处理框架、错误统计
- **配置驱动**：通过配置表而非硬编码，运行时可配置
- **防御性编程**：参数校验、边界检查、空指针保护

### 项目架构

```text
BSP/              # 硬件抽象层（board.h硬件配置隔离）
Core/             # 应用层（main.c，中断处理）
Drivers/          # 驱动层（按类型分类：basic/i2c/spi/display/sensors等）
system/           # 系统服务层（delay、clock_manager、system_init）
Drivers/timer/    # 定时器驱动（TIM2_TimeBase、PWM、输入捕获等）
Debug/            # 调试工具层（debug、log、assert）
common/           # 公共模块（error_code、error_handler）
middlewares/      # 中间件层（协议栈、算法）
Examples/         # 演示案例（example1_led_blink等）
```

### 设计原则

1. **分层隔离**：上层不直接访问底层硬件
2. **配置驱动**：通过配置表而非硬编码
3. **错误处理**：统一错误码与异常处理
4. **可测试性**：模块独立、接口清晰
5. **可移植性**：硬件抽象、跨平台适配

### 已实现模块（分类列表）

**基础驱动模块：**
- GPIO（全模式支持，动态时钟使能）
- LED（配置表驱动，运行时使能/禁用）
- OLED_SSD1306（I2C解耦，支持软硬I2C）
- 硬件I2C（I2C1/I2C2，主模式，总线扫描）
- 软件I2C（任意GPIO引脚，超时支持）
- 硬件SPI、软件SPI
- DS3231（实时时钟，支持软硬I2C）

**系统服务模块：**
- TIM2_TimeBase（1ms中断，动态调频自适应）
- delay（阻塞式+非阻塞式，频率自适应）
- clock_manager（DVFS，9档位，自动调频）
- system_init（统一初始化框架）
- iwdg（独立看门狗管理）
- system_monitor（CPU/内存监控，健康检查）
- module_controller（模块开关总控）

**工具模块：**
- error_handler（统一错误处理框架）
- log（分级日志系统，DEBUG/INFO/WARN/ERROR）

**详细模块列表请参考各模块README和案例代码**

### 系统初始化说明

**System_Init()自动初始化的模块：**
- GPIO模块
- LED模块
- delay模块
- TIM2_TimeBase模块

**需要手动初始化的模块：**
- OLED模块（需要调用`OLED_Init()`）
- clock_manager模块（需要调用`CLKM_Init()`）
- 其他新增模块（根据模块要求初始化）

## ⚙️ 第二部分：核心规范

### 代码规范

#### 命名规范

**通用命名规则：**

| 类型 | 规则 | 示例 |
|------|------|------|
| 目录 | 首字母大写 | `Core/` `Drivers/` |
| 文件 | 全小写下划线 | `led.c` `uart_dma.c` |
| 函数 | 模块名_功能名，符合`Device_Action`格式 | `LED_On()` `PWM_SetFrequency()` |
| 宏 | 全大写下划线 | `LED_ON` `UART_BAUDRATE` |
| 结构体 | 大写开头+`_t` | `LED_Config_t` |
| 枚举 | 模块名_类型名 | `LED_Status_t` |

**硬件/软件模块命名规则：**

对于同时存在硬件和软件版本的通信接口模块（I2C、SPI），使用统一命名规则：

| 类型 | 文件名 | 初始化函数名 |
|------|--------|--------------|
| 硬件I2C | `i2c_hw.c/h` | `I2C_HW_Init()` |
| 软件I2C | `i2c_sw.c/h` | `I2C_SW_Init()` |
| 硬件SPI | `spi_hw.c/h` | `SPI_HW_Init()` |
| 软件SPI | `spi_sw.c/h` | `SPI_SW_Init()` |

**注意**：只有I2C和SPI需要区分硬件/软件版本，其他模块不需要。

#### 错误处理

- **统一返回类型**：所有模块函数必须返回`error_code_t`，禁止返回`void`或`int`
- **错误码设计**：错误码必须在模块设计阶段同步定义，禁止事后补漏
- **模块基值预留**：每个新模块创建前，先在`error_code.h`预留基值（每个模块预留100个错误码空间），再写代码
- **参数校验**：所有公共函数必须进行参数校验
- **错误处理**：使用统一的错误处理框架（error_handler.c/h）

#### 防御性编程

- **空指针检查**：所有函数必须检查空指针参数
- **边界检查**：数组、缓冲区等必须进行边界检查
- **溢出保护**：数值计算必须防止溢出
- **配置驱动**：通过配置表而非硬编码
- **硬件隔离**：BSP层实现硬件抽象
- **禁止硬编码寄存器**：禁止硬编码寄存器地址，所有硬件操作必须通过GPIO层或相应驱动层

#### 代码格式规范

- **头文件规范**：头文件只声明接口，禁止定义变量（使用`static`+`getter`函数）
- **日志调用规范**：`LOG_xxx`调用必须包含模块名字符串，方便定位（如：`LOG_INFO("LED", "LED initialized")`）
- **文件末尾空行**：文件末尾必须保留空行，否则Keil编译警告

### 架构规范

#### 分层原则

- **禁止跨层调用**：禁止跨层调用（如LED层直接操作OLED），违者架构腐烂
- **模块间通信**：模块间通信用回调或事件总线，禁止直接函数调用
- **分层隔离**：严格遵循分层架构，上层不直接访问底层硬件

#### 模块解耦原则

**核心原则**：模块不应定义其他模块的类型，只应使用。

**规则：**
1. **类型定义归属**：每个模块的类型（枚举、结构体、typedef）应由该模块自己定义
2. **禁止跨模块定义**：模块A不应定义模块B的类型，即使是为了"兼容"或"方便"
3. **条件编译解耦**：如果模块依赖可选模块，应使用条件编译处理，而不是定义占位类型

**详细说明和代码示例见第四部分。**

### 开发流程

#### 标准工作流程

```
1. 参考案例（Examples/）
   ↓
2. 配置硬件（BSP/board.h）
   ↓
3. 编写入口函数（Core/main.c）
   ↓
4. 编写业务代码（Core/main.c）
   ↓
5. 清空编译产物（执行 keilkill.bat）
   ↓
6. 编译下载测试
```

**详细操作步骤请参考案例代码** `Examples/`。

#### 模块创建流程

**模块创建前必须完成：**
1. 在`error_code.h`预留模块基值
2. 在`system/config.h`中添加开关宏（`CONFIG_MODULE_XXX_ENABLED`）
3. 设计错误码（模块设计阶段同步定义，禁止事后补漏）

**代码实现要求：**
- 遵循现有代码规范和架构风格
- 使用统一的错误码系统
- 所有模块函数必须返回`error_code_t`，禁止返回`void`或`int`
- 所有`.c`文件必须用`#if CONFIG_MODULE_XXX_ENABLED`包裹全部代码
- 实现硬件抽象（如需要）
- 提供配置表驱动（如适用）
- 添加防御性编程检查
- 创建对应的Example（如需要）

**模块完成后必须：**
- 更新Keil工程文件，否则链接错误
- 测试案例与模块代码同步提交，覆盖率>80%

#### 案例开发流程

**文件命名：**
- `main_example.c`：主程序入口
- `board.h`或`board_config.h`：硬件配置文件
- `config.h`：模块配置文件
- `README.md`：案例说明文档
- `keilkill.bat`：清理编译产物脚本
- `Examples.uvprojx`：Keil工程文件

**文件创建顺序（严格执行）：**
1. 先确认目标目录状态（使用`list_dir`检查）
2. 创建第一个文件（`main_example.c`），自动创建目录
3. 确认目录和文件创建成功（使用`list_dir`再次确认）
4. 创建其他源文件（`board.h`、`config.h`、`README.md`、`keilkill.bat`）
5. **最后**：提示用户手动复制`Examples.uvprojx`工程文件
6. 删除乱码文件夹（如果存在）

**Keil工程文件（.uvprojx）规则（严格执行）：**

**⚠️ 绝对禁止**：AI不得使用任何工具（包括`write`、`read_file`+`write`等）创建或写入`.uvprojx`文件

**正确流程：**
1. **只能人工复制**（必须）：
   - AI只能提示用户手动从相邻项目复制`.uvprojx`文件
   - AI不得使用`write`工具创建或写入`.uvprojx`文件
   - 用户复制后，AI可以使用`read_file`读取并使用`search_replace`工具修改工程文件（如输出名称、添加源文件等）

2. **禁止AI操作**（严格执行）：
   - ❌ 不允许使用 `write` 工具创建`.uvprojx`文件
   - ❌ 不允许使用 `read_file`+`write` 复制`.uvprojx`文件
   - ✅ 只能提示用户手动复制
   - ✅ 用户复制后，AI可以使用`read_file`读取并使用`search_replace`修改工程文件

3. **允许修改**（用户复制后）：
   - 使用 `search_replace` 工具修改输出名称（`OutputName`）
   - 使用 `search_replace` 工具添加或删除源文件引用
   - 使用 `search_replace` 工具修改其他工程配置

**文件夹创建规则（严格执行）：**
1. **先确认状态**：使用 `list_dir` 检查目标目录
2. **使用 write 工具创建**：`write` 工具会自动创建目录，这是最可靠的方法
3. **确认创建成功**：创建第一个文件后，立即使用 `list_dir` 确认
4. **处理乱码文件夹**：
   - 如果发现乱码文件夹，先确认正确文件夹已创建且完整
   - 然后删除乱码文件夹
   - 使用 PowerShell 的 `Remove-Item -Recurse -Force` 命令

**文件夹命名规则：**
- **⚠️ 优先级规则（与其他规则冲突时以此为准）**：当用户用中文描述创建案例时，AI必须将中文翻译成英文来创建文件夹名称
  - 例如：用户说"创建DS323101_读取时间显示"，应创建为 `DS323101_ReadTimeDisplay` 或类似的英文名称
  - 原因：避免中文路径编码问题，确保跨平台兼容性
  - 如果其他规则与此冲突，以本条为准
- 使用英文文件夹名称时，确保使用正确的UTF-8编码
- 避免使用PowerShell命令直接创建包含中文的文件夹，优先使用 `write` 工具创建文件（会自动创建目录）
- 如果出现乱码文件夹，立即删除并使用正确编码重新创建

**使用方式：**
- 每个案例都是独立的Keil工程，可以直接编译和运行
- 案例目录下有`Examples.uvprojx`工程文件
- 直接打开案例工程编译运行，无需复制文件

**代码要求：**
- 尽量不要包含与案例无关的代码
- 除了案例1（基础LED闪烁），所有案例都默认使用OLED作为显示输出设备

**测试前必须：**
- 执行`keilkill.bat`清空所有编译产物，确保重新编译时不会使用旧的编译缓存

#### 代码审查原则

**分层原则**：代码审查应恪守分层原则，确保模块化开发的底层逻辑。

**审查要点：**
- **业务代码留在案例里面**：案例代码中的业务逻辑（如显示格式、业务流程、特定应用场景）应保留在案例中，不要封装到公共模块
- **公共函数归位下沉**：属于公共功能的代码应封装到`common/`目录，供多个案例复用
- **模块功能封装隔离**：属于模块功能的代码应封装到对应模块（`Drivers/`下的分类目录或`system/`），提供清晰的模块接口

**判断标准：**
- **业务逻辑**：特定案例的显示格式、业务流程、应用场景 → 留在案例中
- **公共函数**：多个案例都会用到的通用功能（如工具函数、算法） → 封装到`common/`
- **模块功能**：硬件驱动、系统服务、可复用的功能模块 → 封装到对应模块

#### 模块初始化顺序

**严格遵循初始化顺序，禁止跳跃：**
```
RCC（时钟配置）
  ↓
GPIO（GPIO初始化）
  ↓
外设（具体外设初始化）
  ↓
应用层（应用逻辑初始化）
```

## ⚠️ 第三部分：重要规则

### 1. 已调试完成的代码修改需征询同意

**规则描述**：对于已经标注为"已调试完成"或"已测试通过"的案例或模块，任何修改都必须先征询用户意见，得到明确同意后才能进行修改。

**适用范围：**
- Examples目录下的案例代码
- Drivers目录下的模块代码
- system目录下的系统模块

**工作流程：**
1. 识别已调试完成的代码
2. 需要修改时先询问用户
3. 等待用户明确同意后才能修改
4. 修改后说明修改内容并提醒重新测试

**例外情况（可以不经询问直接修改）：**
- 修复明显的编译错误
- 修复明显的逻辑错误
- 用户明确要求修改的代码

### 2. 每个案例/测试工程都是独立工程

**规则描述**：每个案例和测试工程都是独立的Keil工程，可以直接编译和运行，无需复制到主工程。

**案例工程特点：**
- 每个案例目录下都有`Examples.uvprojx`工程文件
- 每个案例都有独立的`board.h`和`config.h`配置文件
- 每个案例只包含需要的模块源文件
- 直接打开案例工程编译运行，无需复制文件

**配置文件规则：**
- **主工程配置**：`system/config.h` 是主工程使用的配置文件，主工程默认是空项目（所有模块默认启用）
- **案例工程配置**：每个案例使用自己的`config.h`（位于案例目录下），**不使用主工程的`system/config.h`**
- **配置隔离**：案例工程的`config.h`只定义案例需要的模块，未定义的模块不会被编译和链接
- **包含路径**：案例工程的包含路径中，案例目录（`.`）在最前面，确保优先找到案例的`config.h`
- **系统文件处理**：`system`目录下的文件（如`system_init.c`）使用`#include <config.h>`，优先在包含路径中查找，而不是当前目录

**测试工程特点：**
- 由于STM32F103C8T6只有64KB Flash，所有测试用例一起编译可能导致空间不足
- 测试用例采用独立工程的方式，按模块分组（TestDrivers、TestSystem、TestI2C、TestExamples）
- 每个测试工程目录下都有独立的`.uvprojx`工程文件
- 所有测试源文件都在工程目录下（扁平化结构，完全独立）

**使用方法：**
- 打开案例工程：双击`Examples/exampleN_xxx/Examples.uvprojx`
- 打开测试工程：双击`Tests/TestXXX/TestXXX.uvprojx`
- 直接编译（F7）和下载（F8），无需任何文件复制操作

### 3. Git提交由用户执行

**规则描述**：Git提交操作由用户手动执行，AI助手不执行git commit命令。

### 4. 版本号由用户控制

**规则描述**：所有文件中的版本号（包括代码文件头注释中的`@version`、文档中的版本号等）必须由用户控制，AI助手不得随意修改版本号。

**工作流程：**
1. AI助手完成代码或文档修改后，如需更新版本号，应提示用户并提供建议
2. 用户决定是否更新版本号以及更新到哪个版本
3. AI助手根据用户的指示更新版本号

**例外情况：**
- 如果用户明确要求更新版本号，AI助手可以执行
- 如果用户明确要求"更新文档"且包含版本号更新，AI助手可以执行

## 📚 第四部分：详细说明

### 模块依赖和解耦详细说明

**核心原则**：模块不应定义其他模块的类型，只应使用。

#### 依赖处理规范

**1. 可选依赖模块：**
- 如果模块A可选依赖模块B，模块A应尝试包含模块B的头文件
- 如果模块B未启用，模块A应使用条件编译处理相关结构体
- 不应在模块A中定义模块B的类型（即使是前向声明）

**2. 条件包含头文件：**
```c
/* 条件编译：可选依赖模块 */
#ifdef CONFIG_MODULE_XXX_ENABLED
#if CONFIG_MODULE_XXX_ENABLED
#ifndef XXX_H
#include "xxx.h"
#endif
#endif
#endif
```

**3. 条件定义结构体：**
```c
/* 如果依赖模块启用，定义配置结构体 */
#ifdef XXX_H
typedef struct {
    XXX_Type_t field;  /* 使用依赖模块的类型 */
} MyModule_XXX_Config_t;
#else
/* 依赖模块未启用，使用不完整类型（仅用于编译，不应使用） */
typedef struct MyModule_XXX_Config MyModule_XXX_Config_t;
#endif
```

**4. 使用者责任：**
- 使用硬件I2C模式前，使用者必须先包含`i2c_hw.h`
- 使用软件I2C模式前，使用者必须先包含`i2c_sw.h`
- 文档中应明确说明依赖关系和包含顺序

#### 错误示例

```c
/* ❌ 错误：在ds3231.h中定义i2c.h的类型 */
#ifndef I2C_H
typedef enum {
    I2C_INSTANCE_1 = 0,
    I2C_INSTANCE_2 = 1,
} I2C_Instance_t;
#endif
```

#### 正确示例

```c
/* ✅ 正确：尝试包含i2c.h，如果未启用则使用不完整类型 */
#ifdef CONFIG_MODULE_I2C_ENABLED
#if CONFIG_MODULE_I2C_ENABLED
#ifndef I2C_HW_H
#include "i2c_hw.h"
#endif
#endif
#endif

#ifdef I2C_HW_H
typedef struct {
```

**检查步骤：**
1. **列出使用的函数**：列出案例中调用的所有模块函数
2. **检查函数实现**：打开对应的 `.c` 源文件，检查函数是否有实际实现
3. **识别空函数**：空函数特征为只有 `(void)参数;` 和 `return XXX_OK;`，无实际操作代码
4. **处理空函数**：
   - 如果空函数影响核心功能：使用 `#if 0` 条件编译禁用相关代码
   - 在README中明确说明功能限制
   - 在代码中添加警告注释

- **Timer模块**（`Drivers/timer/timer_pwm.c`）：
  - `PWM_Init()`、`PWM_SetFrequency()`、`PWM_SetDutyCycle()`

## 🔧 第五部分：操作规则与历史问题

### 操作规则

#### 禁止操作清单

**1. 禁止AI创建或写入 .uvprojx 文件**
- **绝对禁止**：AI不得使用任何工具创建或写入`.uvprojx`文件
- **正确做法**：AI只能提示用户手动从相邻项目复制`.uvprojx`文件，用户复制后AI可以使用`search_replace`修改工程文件
- **违反后果**：浪费AI计算资源、浪费用户流量、无法完成任务

**2. 禁止在文件夹未创建成功时写文件**
- **绝对禁止**：在文件夹未确认创建成功前，禁止创建任何文件
- **正确流程**：
  1. 先使用 `list_dir` 确认目标目录状态
  2. 使用 `write` 工具创建第一个文件（会自动创建目录）
  3. 使用 `list_dir` 再次确认目录和文件创建成功
  4. 继续创建其他文件
- **违反后果**：文件写入失败，浪费资源，任务无法完成

**3. 禁止复制失败后直接重写**
- **绝对禁止**：当文件复制失败时，禁止直接使用 `write` 工具重写整个文件
- **正确做法**：
  1. 先诊断复制失败的原因（路径问题、权限问题、编码问题）
  2. 解决根本问题（使用正确的路径、请求权限、使用正确的编码）
  3. 重新尝试复制
  4. 只有在确认无法复制且用户明确要求时，才考虑其他方案
- **违反后果**：效率极低，违反规则，浪费资源

#### 必须操作清单

**⚠️ 优先级规则（与其他规则冲突时以此为准）**

**1. 案例文件夹命名必须使用英文**
- **必须执行**：当用户用中文描述创建案例时，AI必须将中文翻译成英文来创建文件夹名称
- **示例**：
  - 用户说："创建DS323101_读取时间显示"
  - AI应创建：`DS323101_ReadTimeDisplay` 或 `DS323101_ReadTime` 等英文名称
- **原因**：
  - 避免中文路径编码问题（PowerShell、文件系统等）
  - 确保跨平台兼容性
  - 减少乱码文件夹的产生
- **优先级**：如果其他规则（如"使用中文文件夹名称"）与此冲突，以本条为准
- **翻译原则**：
  - 保持原意，使用简洁的英文
  - 使用驼峰命名或下划线分隔
  - 避免过长的名称

### 历史问题记录

#### 问题1：文件夹乱码
- **问题**：创建包含中文的文件夹时出现乱码
- **原因**：PowerShell在处理中文路径时的编码问题
- **解决方案**：
  1. **优先方案**：将中文翻译成英文创建文件夹（见"必须操作清单"规则1）
  2. **备选方案**：如果必须使用中文，使用 `write` 工具直接创建文件，工具会自动创建正确的目录结构
- **预防**：
  1. 优先使用英文文件夹名称（将用户的中文描述翻译成英文）
  2. 永远使用 `write` 工具创建文件，不要使用 PowerShell 命令创建目录

#### 问题2：Keil工程文件缺失
- **问题**：创建项目时忘记提示用户手动复制 `Examples.uvprojx`
- **解决方案**：创建案例时必须提示用户手动从相邻项目复制工程文件
- **预防**：在文件创建顺序中明确列出，提示用户手动复制

#### 问题4：文件夹未创建成功就写文件
- **问题**：在文件夹未确认创建成功时就开始写文件
- **原因**：没有先确认状态，盲目操作
- **后果**：文件写入失败，浪费资源
- **解决方案**：
  1. 每次创建文件前，先使用 `list_dir` 确认目录状态
  2. 创建第一个文件后，立即确认创建成功
  3. 确认成功后再继续创建其他文件
