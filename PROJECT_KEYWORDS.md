### 核心特性

- **硬件抽象**：BSP层隔离，board.h配置表驱动，跨型号移植便捷
- **模块化架构**：清晰的分层设计（BSP/驱动/系统/应用），职责分离
- **统一错误处理**：错误码系统、错误处理框架、错误统计
- **配置驱动**：通过配置表而非硬编码，运行时可配置
- **防御性编程**：参数校验、边界检查、空指针保护

### 项目架构

```text
BSP/              # 硬件抽象层（board.h硬件配置隔离）
Core/             # 应用层（main.c，中断处理）
Drivers/          # 驱动层（按类型分类：basic/i2c/spi/display/sensors等）
system/           # 系统服务层（delay、clock_manager、system_init）
Drivers/timer/    # 定时器驱动（TIM2_TimeBase、PWM、输入捕获等）
Debug/            # 调试工具层（debug、log、assert）
common/           # 公共模块（error_code、error_handler）
middlewares/      # 中间件层（协议栈、算法）
Examples/         # 演示案例（example1_led_blink等）
```

### 设计原则

1. **分层隔离**：上层不直接访问底层硬件
2. **配置驱动**：通过配置表而非硬编码
3. **错误处理**：统一错误码与异常处理
4. **可测试性**：模块独立、接口清晰
5. **可移植性**：硬件抽象、跨平台适配

### 代码规范

#### 命名规范 `[通用]` `[P1]`

**通用命名规则：**

| 类型 | 规则 | 示例 |
|------|------|------|
| 目录 | 首字母大写 | `Core/` `Drivers/` |
| 文件 | 全小写下划线 | `led.c` `uart_dma.c` |
| 函数 | 模块名_功能名，符合`Device_Action`格式 | `LED_On()` `PWM_SetFrequency()` |
| 宏 | 全大写下划线 | `LED_ON` `UART_BAUDRATE` |
| 结构体 | 大写开头+`_t` | `LED_Config_t` |
| 枚举 | 模块名_类型名 | `LED_Status_t` |

**硬件/软件模块命名规则：**

对于同时存在硬件和软件版本的通信接口模块（I2C、SPI），使用统一命名规则：

| 类型 | 文件名 | 初始化函数名 |
|------|--------|--------------|
| 硬件I2C | `i2c_hw.c/h` | `I2C_HW_Init()` |
| 软件I2C | `i2c_sw.c/h` | `I2C_SW_Init()` |
| 硬件SPI | `spi_hw.c/h` | `SPI_HW_Init()` |
| 软件SPI | `spi_sw.c/h` | `SPI_SW_Init()` |

**注意**：只有I2C和SPI需要区分硬件/软件版本，其他模块不需要。

#### 错误处理

- **统一返回类型** `[通用]` `[P0]`：所有模块函数必须返回`error_code_t`，禁止返回`void`或`int`
- **错误码设计** `[通用]` `[P1]`：错误码必须在模块设计阶段同步定义，禁止事后补漏
- **模块基值预留** `[通用]` `[P1]`：每个新模块创建前，先在`error_code.h`预留基值（每个模块预留100个错误码空间），再写代码
- **参数校验** `[通用]` `[P1]`：所有公共函数必须进行参数校验
- **错误处理** `[通用]` `[P1]`：使用统一的错误处理框架（error_handler.c/h）

#### 模块函数返回类型规范 `[通用]` `[P0]`

**强制要求**：
- **所有公共函数必须返回`error_code_t`**，包括：
  - `Init()` 函数
  - `Deinit()` 函数
  - `Set*()` 函数
  - `Get*()` 函数
  - `Enable*()` / `Disable*()` 函数
  - 所有其他公共接口函数

**禁止的返回类型**：
- ❌ `void` - 禁止返回void（无法传递错误信息）
- ❌ `int` - 禁止返回int（应使用`error_code_t`）
- ❌ `bool` - 禁止返回bool（应使用`error_code_t`，可以定义`XXX_OK`和`XXX_ERROR`）

**历史遗留问题处理**：
- 对于已存在的返回`void`的函数（如`TIM2_TimeBase_Init()`），应：
  1. 在模块文档中明确标注为"历史遗留，待重构"
  2. 新模块禁止使用`void`返回类型
  3. 重构时优先处理这些函数

**示例**：
```c
/* ✅ 正确：返回error_code_t */
error_code_t LED_Init(void);
error_code_t PWM_SetFrequency(PWM_Instance_t instance, uint32_t freq);

/* ❌ 错误：返回void */
void TIM2_TimeBase_Init(void);  /* 历史遗留，待重构 */

/* ❌ 错误：返回int */
int GPIO_ReadPin(GPIO_TypeDef *port, uint16_t pin);  /* 应返回GPIO_Status_t */
```

#### 占位空函数规范 `[通用]` `[P1]`

**定义**：占位空函数（placeholder functions）是指功能未实现、只有函数框架的函数，用于保持接口完整性，待后续完善。

**强制要求**：

1. **必须返回错误码** `[通用]` `[P0]`：
   - 占位空函数**禁止返回成功码**（如`XXX_OK`）
   - 必须返回特定的"未实现"错误码（如`XXX_ERROR_NOT_IMPLEMENTED`）
   - 这样使用者检查返回值时能立即发现功能未实现
   - AI在开发时能通过错误处理发现需要完善的函数

2. **错误码定义**：
   - 在模块错误码枚举中必须定义`XXX_ERROR_NOT_IMPLEMENTED`
   - 建议使用模块基值-99（如`ERROR_BASE_TIMER - 99`）
   - 全局定义：`error_code.h`中已定义`ERROR_NOT_IMPLEMENTED = -9999`作为参考

3. **函数注释**：
   - 必须在函数注释中明确标注`@note 占位空函数，功能未实现，待完善`
   - 必须在函数体中添加注释说明

4. **编译时警告**（推荐）：
   - 建议在占位空函数中添加`#warning`编译时警告
   - 这样编译时就能发现需要完善的函数

5. **运行时日志**（可选）：
   - 如果启用了日志模块，建议添加日志输出
   - 使用`LOG_WARN`级别输出警告信息

**标准实现格式**：

```c
/**
 * @brief 设置PWM频率
 * @param[in] instance PWM实例索引
 * @param[in] frequency 频率（Hz）
 * @return PWM_Status_t 错误码
 * @note 占位空函数，功能未实现，待完善
 */
PWM_Status_t PWM_SetFrequency(PWM_Instance_t instance, uint32_t frequency)
{
    (void)instance;
    (void)frequency;
    
    /* 编译时警告 */
    #warning "PWM_SetFrequency: 占位空函数，功能未实现，待完善"
    
    /* 运行时日志（可选） */
    #if CONFIG_MODULE_LOG_ENABLED
    LOG_WARN("PWM", "PWM_SetFrequency: 占位空函数，功能未实现");
    #endif
    
    /* ⚠️ 占位空函数：功能未实现，待完善 */
    return PWM_ERROR_NOT_IMPLEMENTED;  /* 返回错误码，而不是PWM_OK */
}
```

**错误码定义示例**：

```c
/* 在模块错误码枚举中 */
typedef enum {
    PWM_OK = ERROR_OK,
    PWM_ERROR_NOT_IMPLEMENTED = ERROR_BASE_TIMER - 99,  /* 功能未实现（占位空函数） */
    PWM_ERROR_NOT_INITIALIZED = ERROR_BASE_TIMER - 1,
    PWM_ERROR_INVALID_PARAM = ERROR_BASE_TIMER - 2,
    // ...
} PWM_Status_t;
```

**AI开发时的处理**：

1. **识别占位空函数**：
   - 编译时看到`#warning`警告
   - 运行时检查返回值发现`XXX_ERROR_NOT_IMPLEMENTED`
   - 查看函数实现发现只有`return XXX_ERROR_NOT_IMPLEMENTED`

2. **完善占位空函数**：
   - 当AI需要实现功能时，应完善所有占位空函数
   - 完善后移除`#warning`和占位注释
   - 将返回值改为实际实现后的错误码

3. **案例开发时的处理**：
   - 如果案例使用了占位空函数，必须在README中说明
   - 使用`#if 0`条件编译禁用相关代码（如果影响核心功能）
   - 在代码中添加警告注释

**已知占位空函数模块**：
- **Timer模块**（`Drivers/timer/timer_pwm.c`）：大部分函数为占位空函数
- **DMA模块**（`Drivers/dma/dma.c`）：大部分函数为占位空函数

#### 模块初始化顺序 `[通用]` `[P1]`

**严格遵循初始化顺序，禁止跳跃：**
```
RCC（时钟配置）
  ↓
GPIO（GPIO初始化）
  ↓
外设（具体外设初始化）
  ↓
应用层（应用逻辑初始化）
```

**详细说明**：

1. **RCC（时钟配置）**：
   - 系统启动时自动配置（在`system_stm32f10x.c`中）
   - 如需修改时钟配置，在`SystemInit()`中修改
   - 时钟管理模块（clock_manager）可以动态调整频率

2. **GPIO（GPIO初始化）**：
   - 通常由`System_Init()`自动初始化
   - 或由具体模块初始化时自动配置（如LED模块初始化时会配置LED的GPIO）

3. **外设（具体外设初始化）**：
   - 通信接口（I2C、SPI、UART等）初始化
   - 定时器、ADC、DAC等外设初始化
   - 传感器模块（DS3231、MAX31856等）初始化
   - **注意**：外设初始化前，必须先初始化其依赖的通信接口（如DS3231使用I2C，必须先初始化I2C）

4. **应用层（应用逻辑初始化）**：
   - 业务逻辑初始化
   - 状态机初始化
   - 应用数据结构初始化

**初始化示例**：
```c
int main(void)
{
    /* 1. 系统初始化（自动初始化GPIO、LED、delay、TIM2_TimeBase） */
    System_Init();
    
    /* 2. 初始化其他模块（按依赖顺序） */
    /* 2.1 初始化通信接口 */
    I2C_HW_Init(I2C_INSTANCE_2);  /* 先初始化I2C */
    
    /* 2.2 初始化依赖通信接口的模块 */
    DS3231_Config_t ds3231_config;
    ds3231_config.interface_type = DS3231_INTERFACE_HARDWARE;
    ds3231_config.config.hardware.i2c_instance = I2C_INSTANCE_2;
    DS3231_Init(&ds3231_config);  /* 再初始化DS3231（依赖I2C） */
    
    /* 2.3 初始化显示模块 */
    OLED_Init();  /* OLED初始化（可能依赖I2C或软件I2C） */
    
    /* 3. 应用逻辑初始化 */
    /* 业务逻辑初始化代码 */
    
    /* 4. 主循环 */
    while (1)
    {
        /* 业务逻辑 */
    }
}
```

**常见错误**：
- ❌ 在I2C初始化前初始化DS3231（DS3231依赖I2C）
- ❌ 在GPIO初始化前初始化LED（LED依赖GPIO）
- ❌ 跳过依赖模块直接初始化上层模块

## 📚 第四部分：详细说明

### 模块依赖和解耦详细说明

**核心原则**：模块不应定义其他模块的类型，只应使用。

#### 依赖处理规范

**1. 可选依赖模块：**
- 如果模块A可选依赖模块B，模块A应尝试包含模块B的头文件
- 如果模块B未启用，模块A应使用条件编译处理相关结构体
- 不应在模块A中定义模块B的类型（即使是前向声明）

**2. 条件包含头文件：**
```c
/* 条件编译：可选依赖模块 */
#ifdef CONFIG_MODULE_XXX_ENABLED
#if CONFIG_MODULE_XXX_ENABLED
#ifndef XXX_H
#include "xxx.h"
#endif
#endif
#endif
```

**3. 条件定义结构体：**
```c
/* 如果依赖模块启用，定义配置结构体 */
#ifdef XXX_H
typedef struct {
    XXX_Type_t field;  /* 使用依赖模块的类型 */
} MyModule_XXX_Config_t;
#else
/* 依赖模块未启用，使用不完整类型（仅用于编译，不应使用） */
typedef struct MyModule_XXX_Config MyModule_XXX_Config_t;
#endif
```

**4. 使用者责任：**
- 使用硬件I2C模式前，使用者必须先包含`i2c_hw.h`
- 使用软件I2C模式前，使用者必须先包含`i2c_sw.h`
- 文档中应明确说明依赖关系和包含顺序

#### 错误示例

```c
/* ❌ 错误：在ds3231.h中定义i2c.h的类型 */
#ifndef I2C_H
typedef enum {
    I2C_INSTANCE_1 = 0,
    I2C_INSTANCE_2 = 1,
} I2C_Instance_t;
#endif
```

#### 正确示例

```c
/* ✅ 正确：尝试包含i2c.h，如果未启用则使用不完整类型 */
#ifdef CONFIG_MODULE_I2C_ENABLED
#if CONFIG_MODULE_I2C_ENABLED
#ifndef I2C_HW_H
#include "i2c_hw.h"
#endif
#endif
#endif

#ifdef I2C_HW_H
typedef struct {
    I2C_Instance_t i2c_instance;  /* 使用i2c.h定义的类型 */
} DS3231_HardwareI2C_Config_t;
#else
typedef struct DS3231_HardwareI2C_Config DS3231_HardwareI2C_Config_t;
#endif
```

**设计目标：**
- ✅ 模块职责清晰，各司其职
- ✅ 依赖关系明确，易于维护

##### 案例创建规范

创建新案例时，必须检查案例中使用的模块函数是否为**占位空函数**（placeholder functions）。占位空函数虽然可以编译通过，但不会执行实际功能，可能导致案例无法正常工作。

**检查步骤：**
1. **列出使用的函数**：列出案例中调用的所有模块函数
2. **检查函数实现**：打开对应的 `.c` 源文件，检查函数是否有实际实现
3. **识别占位空函数**：
   - **特征1**：函数体只有 `(void)参数;` 和 `return XXX_ERROR_NOT_IMPLEMENTED;`
   - **特征2**：函数注释中包含`@note 占位空函数`或`待实现`
   - **特征3**：编译时有`#warning`警告提示
   - **特征4**：运行时检查返回值发现`XXX_ERROR_NOT_IMPLEMENTED`
4. **处理占位空函数**：
   - 如果占位空函数影响核心功能：使用 `#if 0` 条件编译禁用相关代码
   - 在README中明确说明功能限制
   - 在代码中添加警告注释
   - 检查返回值，如果返回`XXX_ERROR_NOT_IMPLEMENTED`，说明功能未实现

**已知的占位空函数模块：**
- **Timer模块**（`Drivers/timer/timer_pwm.c`）：

- **DMA模块**（`Drivers/dma/dma.c`）：大部分函数为占位空函数
- **CAN模块**（`Drivers/can/can.c`）：大部分函数为占位空函数

  4. 只有在确认无法复制且用户明确要求时，才考虑其他方案

#### Q4: 案例工程和主工程的配置文件有什么区别？

#### Q8: 如何检查案例中使用的函数是否是占位空函数？
**检查步骤**：
1. **列出使用的函数**：列出案例中调用的所有模块函数
2. **检查函数实现**：打开对应的 `.c` 源文件，检查函数是否有实际实现
3. **识别占位空函数**：
   - **特征1**：函数体只有 `(void)参数;` 和 `return XXX_ERROR_NOT_IMPLEMENTED;`
   - **特征2**：函数注释中包含`@note 占位空函数`或`待实现`
   - **特征3**：编译时有`#warning`警告提示
   - **特征4**：运行时检查返回值发现`XXX_ERROR_NOT_IMPLEMENTED`
4. **处理占位空函数**：
   - 如果占位空函数影响核心功能：使用 `#if 0` 条件编译禁用相关代码
   - 在README中明确说明功能限制
   - 在代码中添加警告注释
   - 检查返回值，如果返回`XXX_ERROR_NOT_IMPLEMENTED`，说明功能未实现

**相关规则**：[占位空函数规范](#占位空函数规范-通用-p1)、[案例创建规范](#案例创建规范)

#### Q13: 模块初始化失败如何处理？

2. **错误处理**：使用统一的错误处理框架（error_handler.c/h）处理错误
3. **错误码查找**：根据错误码查找具体原因（参考`error_code.h`和模块文档）

#### Q14: 如何添加新的错误码？
**添加步骤**：
1. **确定模块基值**：在`error_code.h`中查找模块对应的基值（如`ERROR_BASE_LED = -400`）
2. **定义错误码**：在模块头文件中定义错误码枚举，从基值开始递减
   ```c
   typedef enum {
       LED_OK = ERROR_BASE_LED,                    // -400
       LED_ERROR_INVALID_PARAM = ERROR_BASE_LED - 1,  // -401
       LED_ERROR_INIT_FAILED = ERROR_BASE_LED - 2,    // -402
       // ... 最多100个错误码空间
   } LED_Status_t;
   ```
3. **注册到错误处理**：在`error_handler.c`的`ErrorModule_Init()`中注册模块（如已注册可跳过）
4. **使用错误码**：在模块函数中返回对应的错误码

**注意事项**：
- 错误码必须在模块设计阶段同步定义，禁止事后补漏
- 每个模块预留100个错误码空间，不要超出范围
- 错误码从基值开始递减定义

**相关规则**：[错误处理 - 错误码设计](#错误处理)、[错误处理 - 模块基值预留](#错误处理)

#### Q15: 配置表驱动的具体实现方法？

```c
#define LED_CONFIGS { \
    {GPIOA, GPIO_Pin_1, Bit_RESET, 1}, \
```

```c
static LED_Config_t led_configs[] = LED_CONFIGS;

```

**优势**：
- 硬件配置集中管理，便于修改
- 支持多实例配置
- 运行时可通过配置表控制功能

### 开发术语

| 术语 | 说明 |
|------|------|
| **独立工程** | 每个案例都是独立的Keil工程，可直接编译运行 |
| **配置隔离** | 案例工程使用自己的config.h，不依赖主工程配置 |
| **防御性编程** | 参数校验、边界检查、空指针保护等编程实践 |
| **模块基值** | error code base，每个模块在error_code.h中预留的错误码起始值 |
| **错误码空间** | 每个模块预留100个错误码空间，从基值开始递减定义 |
| **文件创建顺序** | 创建案例时必须遵循的文件创建顺序，确保目录正确创建 |
| **占位空函数检查** | 创建案例时必须检查使用的函数是否为占位空函数 |
| **代码审查原则** | 判断代码应该放在案例、common还是模块目录的原则 |
| **模块初始化顺序** | 严格遵循RCC → GPIO → 外设 → 应用层的初始化顺序 |

### 模块创建检查清单

**创建前**：
- [ ] 在`error_code.h`预留模块基值
- [ ] 在`system/config.h`中添加开关宏
- [ ] 设计错误码（模块设计阶段同步定义）

**实现中**：
- [ ] 所有函数返回`error_code_t`
- [ ] 所有`.c`文件用`#if CONFIG_MODULE_XXX_ENABLED`包裹
- [ ] 实现硬件抽象（如需要）
- [ ] 提供配置表驱动（如适用）
- [ ] 添加防御性编程检查

**完成后**：
- [ ] 更新Keil工程文件
- [ ] 测试案例与模块代码同步提交（覆盖率>80%）

### 案例创建检查清单

**文件创建顺序**：
1. [ ] 使用`list_dir`确认目标目录状态
2. [ ] 创建第一个文件（`main_example.c`）
3. [ ] 使用`list_dir`确认创建成功
4. [ ] 创建其他源文件（`board.h`、`config.h`、`README.md`、`keilkill.bat`）
5. [ ] 提示用户手动复制`Examples.uvprojx`
6. [ ] 删除乱码文件夹（如果存在）

**代码要求**：
- [ ] 检查使用的函数是否为占位空函数（检查返回值是否为`XXX_ERROR_NOT_IMPLEMENTED`）
- [ ] 如果使用了占位空函数，在README中说明功能限制
- [ ] 尽量不要包含与案例无关的代码
- [ ] 除了案例1，所有案例默认使用OLED作为显示输出设备

**测试前**：
- [ ] 执行`keilkill.bat`清空编译产物

### 提交前检查清单

- [ ] 换行符格式（统一使用LF或CRLF）
- [ ] 返回值类型（必须为`error_code_t`）

- [ ] 文件末尾空行
- [ ] 错误码已定义（`error_code.h`中已添加）
